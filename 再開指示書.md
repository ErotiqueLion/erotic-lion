## 4. 重要な実装上の注意事項

### プロンプト構造（必ず守ること）
```
system_instruction  → ゲームロジック・JSON スキーマのみ（クリーンな内容）
ユーザーメッセージ冒頭 → [NPC設定: ${char.prompt} 現在の欲情度: ${arousal}%]
```
**エロ系プロンプトを `system_instruction` に入れると空レスポンスになる。絶対にやらないこと。**

### AIが返すJSONスキーマ（PC版）
```json
{
  "valid": true/false,
  "player_lost": true/false,
  "sister_lost": true/false,
  "word": "返答の単語",
  "word_reading": "よみがな",
  "feedback": "表示用セリフ（漢字あり）",
  "tts_text": "TTS用セリフ（漢字→よみがな に置換済み）",
  "tts_instruction": "感情指示",
  "arousal_delta": 数値
}
```

### Gemini TTS のクォータ
- モデル: `gemini-2.5-flash-preview-tts`
- 無料枠: **1日100リクエスト**（5〜10回プレイで枯渇）
- 推奨: 設定画面の優先エンジンを **「Google Cloud」** に設定して運用する
- 課金を有効化すれば上限が緩和される

### iOS Safari 対応（モバイル版のみ）
- `new Audio().play()` は非同期コンテキストでブロックされる → `onended` が発火せずフリーズ
- **解決策**: ユーザージェスチャー時に `AudioContext` を初期化し、以後はそれで再生
- マイク初回タップ: iOS がパーミッションダイアログを出す際に `'aborted'` エラーが発生 → 300ms 後に1回だけリトライ
- **PC版は変更しないこと**（`src/App.jsx` に AudioContext コードを混入させない）

### しりとり検証の3層構造
1. **クライアント事前チェック**（`handlePlayerInput` 冒頭）: 既出 / 末尾「ん」 / かな入力時の先頭文字
2. **AI側検証**（`getSystemPrompt` 内のルール）: `valid` / `player_lost` / `sister_lost` フラグ
3. **AI返答事後検証**（`JSON.parse` 後）: 既出 / 読みミスマッチリトライ / 読み末尾「ん」

### API キー（両方 Google Cloud Console から取得）
| キー | 用途 |
|---|---|
| `geminiApiKey` | Generative Language API（テキスト生成 + Gemini TTS） |
| `gcpApiKey` | Cloud Text-to-Speech API（Neural2） |

## 6. 開発環境

```bash
# PC版
cd C:\ErotiqueLion\EroticWordChain
npm run dev    # ローカル開発サーバー起動（localhost:5173）
npm run build  # 本番ビルド → dist/ に出力

# モバイル版
cd C:\ErotiqueLion\EroticWordChain\mobile
npm run dev    # ローカル開発サーバー起動
npm run build  # 本番ビルド → mobile/dist/ に出力

git push origin main  # リモートへ反映（PC版 + モバイル版 + kai版 自動ビルド）
```

> [!IMPORTANT]
> 再開時は必ず `localStorage` に有効な Google Cloud Console の API キーが保存されているか確認すること。
> コンソールの `Gemini null reason:` / `Gemini TTS no audio:` / `AI reading mismatch:` ログで問題の原因を特定できる。

> [!NOTE]
> - `src/App_bk.jsx` は旧バージョンのバックアップ。参照用であり編集不要。
> - `kai/` は kuroshiro 実験版。修正対象外。
> - **PC版 (`src/`) と モバイル版 (`mobile/`) は独立コードベース。PCの修正がモバイルに自動反映されるわけではない。**
